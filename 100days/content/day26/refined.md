Day 26 Clawdbot 之亂

最近這兩天瘋狂被 Clawdbot 洗版，可以說是 Clawdbot 之亂了，來發篇逆風的看法。

有一些吹捧說可以取代 n8n 的文，這樣的比較其實偏怪，因為本質上定位不同，n8n 強大之處在於有既定的 workflow 可以把這些流程串接起來，是應用場景的創造工具，而 Clawdbot 是在裝置上幫你派發任務的 Adapter，讓你可以輕鬆調用不同的應用場景，意思是 n8n 是客服，而 Clawdbot 是總機。

從 GitHub Star 數（https://github.com/clawdbot/clawdbot）來看，Clawdbot 是在最近幾天大爆發，跟社群討論度非常相關，應該是一些人的吹捧分享，造成這個名字成為 Buzzword 的正向飛輪所致。實際安裝起來，根本不是那麼實用，至少我覺得對大部分人而言還是如此。官方在安裝的流程上有提供已經整合的工具清單，看了一圈真的沒有幾個實用的，設定起來也還是會卡卡的。Clawdbot 的優勢在讓你用平常習慣的通訊軟體就可以操作 AI，不能否認這的確是便利且需要的功能，但同時又有多少人真的有這樣的需求？人在外面臨時要 AI 回信？要 AI 操作行事曆？認真？這種特定用途的工具不是都已經有官方提供的 App 了？ Google 的整合已經很高了，如果還沒習慣用代表你也更不會習慣用 Clawdbot，而且你還得先串接好能讓 AI 操作回信什麼的設定。

目前在我身上的遠端用途就是寫程式相關的需求，那其實 Claude Code + Happy 就已經很足夠，目前美中不足的是還沒支援 Slash Command，而且 Claude Code 其實可以不只操作當前的目錄，所以基本上大部分寫程式相關的行為都可以靠它完成了。

其他各式需求其實就只差在 Skills 支援的豐富度，那在這個角度看其實 Claude Code 就已經可以取代 Clawdbot，甚至還沒有複雜的設定，可以透過 Plugins 一鍵安裝 Agent、Skills、MCP，我認為 Clawdbot 只會是一時風潮，過沒多久就沒什麼人討論了。

最後消毒一下，當然不是所有人的分享都那麼沒料，要看作者的實際案例是如何，有沒有分享到解決平常就是你的痛點的流程或是工具的設定，但我相信應該真的只符合少數特定人士。至於「Mac Mini 很適合跑 Clawdbot」這件事，為了 Clawdbot 這個其實真的是偶爾用到的東西買一台設備放在家當 Server 我是真的看不懂，人家大神用盡各種方式把 Claude Max 的額度用完去買一台還情有可原，大部分只是被這波熱潮燒到的人真的是不太需要...，把舊裝置設定一下就可以了沒有非 Mini 不可。

以下開放打臉 Clawdbot 其實超強超棒是我不懂，留言絕不關閉及封鎖，大家別客氣！
